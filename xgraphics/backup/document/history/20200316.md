FILE I/O
========

파일 입출력 중에서 입력과 관련된 부분만 일단 구현하였습니다.
출력은 현재 급하게 필요하지 않기 때문입니다.
관련하여 콘솔 입출력을 리팩터링 해보려 했는데,
이것도 잠시 뒤로 미루어 봅니다.

VECTOR & MATRIX
===============

이제 벡터와 매트릭스를 짜 보려 합니다.
몇몇 함수들은 그리 마음에 들지 않지만,
계속 변경해간다고 생각하면서 개발을 하고 있습니다.
이제 대략 그래픽스 관련 코드로 넘어가려 하네요.

GRAPHICS BASIS
==============

그래픽스의 기본은 벡터와 매트릭스가 아닐까 합니다.
많은 자료 구조를 구현해 놓은 것을 보면 벡터와 매트릭스 자료 구조가 가장 많더라구요.
저는 그래픽스에서 사용할 벡터와 매트릭스를 4차원으로 국한시켰고,
타입을 리얼만 사용하도록 강제하였습니다.
굳이 따로 템플릿을 사용하는 것에 코드 가독성이 떨어지더라구요.
벡터까지 구현했군요.
카피 대입 연산자에 대한 혼동이 있습니다.
이렇게 동작하겠지라고 생각하고 짤 수 있도록 도와주는 것이 라이브러리가 아닐까 합니다.
제약을 많이 두는 것 보다 제약 없이 예상가능한 프로그램을 짤 수 있도록 돕는 것이 정말 좋은 라이브러리가 아닐까요.
그것을 해나가려고 하니 어렵기는 합니다.
여러 제약을 두고 있는 것을 보면서,
그 제약을 없애야 겠지요.
쉐이더에 대한 기본까지 구현하였습니다.
쉐이더가 그래픽스 루틴과 관련한 어떤 로직을 가지고 있기 보다.
프로그램이 가지고 있어야 한다고 생각이 되더군요.

그래픽스를 이제 조금 구현했습니다.
언제나 기반이 잘 다져지면 그 다음부터 조금씩 프로그램을 올리는 일은 수월해 집니다.
모두들 잘 갖춰진 기반 때문에 여러 그래픽스 관련된 혹은 게임 관련된 엔진을 사용하는것 입니다.
그저 엔진이 어떤 구조를 가져야는지에 대한 지적 호기심이죠.

버포, 쉐이더, 프로그램, 어플리케이션, 카메라, 벡터, 매트릭스, 프로젝션, 광원 버텍스와 버티시즈.
이렇게 구현했습니다.
이제 가장 중요한 그래픽스 객체를 정의하고 움직이려 합니다.
어찌했건 물리와 엮여야 하는 것 같아서

X::PHYSIC 란 기본 물리와 그래픽스에서 사용되어질 물리를 X::GRAPHICS::PHYSIC 란 이름 공간으로 나눌 생각입니다.

객체와 레퍼런스란 개념을 둘 것인데, 하나의 버텍스 정보를 찰지게 재활용할 생각이기에 고민을 하고 접근할 생각입니다.
처음부터 어느 정도 고민하고 가도 계속 리팩터링꺼리가 나옵니다.
다만 줄어들기는 하죠.
